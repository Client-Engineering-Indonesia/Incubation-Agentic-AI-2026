## Understanding Environments

### What are Environments?

The watsonx Orchestrate ADK introduces **environments (env)**, which represent instances of watsonx Orchestrate you can access.

**Environments can be:**

1. **Local**: Running on your laptop using the Developer Edition of watsonx Orchestrate
2. **Remote**: Hosted on IBM Cloud, AWS, or on-premises using Cloud Pak for Data

### Environment Behavior

When you **activate an environment**, all commands (except `orchestrate server` and `orchestrate chat`) target that environment. This includes:
- Importing agents, tools, and connections
- Listing resources
- Removing resources
- Deploying agents

## Part 2: Configure Access to Remote Environments

### Step 2: Add Remote Environment

Use the `orchestrate env add` command to add a remote environment to your local CLI.

**Syntax:**
```bash
orchestrate env add -n <environment-name> -u <your-instance-url>
```

**Example:**
```bash
orchestrate env add \
  -n workshop \
  -u https://us-south.ml.cloud.ibm.com/ml/v1/instances/abc123-def456-ghi789
```

**Parameter Explanation:**
- `-n workshop`: Name for this environment (you choose this)
- `-u <url>`: Service instance URL you copied in Step 1

---

### Step 3: Activate Environment

Use the `orchestrate env activate` command to authenticate against the environment and target all commands to it.

**Syntax:**
```bash
orchestrate env activate <environment-name>
```

**Example:**
```bash
orchestrate env activate workshop
```

**Interactive Prompt:**
```
Please enter WXO API key: 
```

**Enter the API key** you generated in Step 1.

You will see your local enviornment has been activated.

```
[INFO] - Environment 'workshop' is now active
```
